📋 Cybersalt MCP for Joomla — Changelog

v1.8.0 2026-05-23

cs-mcp-for-j now manages a second Joomla extension end-to-end. The new RSTicketsPro MCP add-on adds 20 typed tools for RSJoomla!'s helpdesk extension, alongside three new 4SEO Business Profile tools that complete the LocalBusiness JSON-LD picture for sites running 4SEO. Tool count 80 → 103. Domain count 13 → 14.

📦New Features

  • RSTicketsPro MCP add-on (plg_system_csmcpforjrst) — a new bundled-but-separable plugin that brings the RSJoomla! RSTicketsPro 3.x helpdesk extension under MCP control. 20 tools: 6 ticket reads (list_rst_tickets with JOIN'd labels and rich filters, get_rst_ticket with autoclose ETA, get_rst_ticket_messages with decoded system events, plus history/notes/files), 6 lookups (departments / statuses / priorities / staff / groups / custom_fields), and 8 writes (add_rst_ticket_reply, add_rst_ticket_note, update_rst_ticket with from/to diff, close_rst_ticket (also stops time tracking), reopen_rst_ticket, flag_rst_ticket, notify_rst_ticket, delete_rst_ticket requires confirm:true). Write tools call into RST's own AdminModel so every email notification, ticket_history audit, dept-change code regeneration, and staff-access validation happens automatically.
  • 4SEO Business Profile typed wrappers (get_4seo_business_profile, set_4seo_business_profile, clear_4seo_business_profile) — site-wide LocalBusiness JSON-LD profile (the #defaultBusiness node 4SEO emits on every page). Handles the structural gotchas (single-element array for type + country, logo normalised, 28-field hours grid from a clean [{day, opens, closes}]), and creates the row from scratch when the human hasn't saved the admin form yet. The previously-only working LocalBusiness path on Cybersalt client sites.

🔧Improvements

  • get_rst_ticket_messages decodes system messages from RST's serialize()'d PHP blobs into clean system_event {type, from, to, user_id} objects. Uses allowed_classes => false on the unserialize() call so no PHP gadget chain can ever instantiate. New include_system_messages arg to hide them when you want just the conversation thread.
  • get_rst_ticket includes an autoclose block — answers "when will this ticket autoclose?" with computed warning/close ETAs + a blocked_by field when conditions short-circuit the flow.
  • Shared withSiteAppContext() trait helper — extracted SiteApplication-swap pattern reused across every RST write tool that fires emails (Reply / Update / Close / Reopen / Notify).

🐛Bug Fixes

  • add_rst_ticket_reply failed with "Error loading menu: api" in API context. RST's reply flow assumes a SiteApplication; the api app has no menu/router. Fix bypasses reply() and calls saveMessage() directly, wrapped in withSiteAppContext() swap. Validated end-to-end on virtuemarttemplates.net TECH-0000000202. Full investigation at resolved-issues/ISSUE-5-add_rst_ticket_reply-site-app-context.md.
  • list_rst_tickets / get_rst_ticket returned staff_name=null — JOIN went through _rsticketspro_staff.id but tickets.staff_id is actually a Joomla user_id (confirmed in models/fields/staff.php). Direct users.id JOIN now resolves correctly.
  • Update / Close / Reopen / AddReply returned stale post-write state — RST's getTicket() caches statically and write methods don't invalidate the cache. New fetchTicketRow() trait helper reads fresh via direct SQL.
  • update_rst_ticket changes array was always empty for staff/dept changes — updateInfo() calls $original->bind($data) mid-flight which mutates the snapshot. Fix: snapshot scalar values BEFORE the update call, not the JTable reference.
  • notify_rst_ticket always returned notified:true regardless of whether RST actually sent the email. Now captures the model return value + emits an explanatory note.

⚠️Known limitations carried forward

  • File attachments still not supported via add_rst_ticket_reply — use the admin reply box for replies that need attachments.
  • tickets.staff_id naming trap — column stores a Joomla user_id despite the name. list_rst_staff output disambiguates with both user_id (use this for assignments) and staff_id (the _staff table PK).

v1.7.5 2026-05-13

Three field-discovery fixes from the Westshore Eye Care SEO audit. Tool count 78 → 80.

🐛Bug Fixes

  • set_4seo_meta_override wasn't flipping the data.use* flags. The tool was writing custom.title / description / etc. correctly and setting status_title/status_description to 2, but 4SEO's renderer also checks data.useTitle / useDescription / useRobots / useCanonical at request time — and those were stuck at 0 since v1.7.0. So overrides were silently inert despite looking right in the DB. Both insert and update branches now flip the matching flag whenever a custom value is supplied. Response includes use_flags_set: [...]. 4SEO Free sites: per-page custom meta appears to be Pro-only — if writes look clean but pages don't change, that's almost certainly the cause.
  • Dashboard token preview now shows the substituted token live, not just at copy-click time. The <code> block re-renders on paste / clear / refresh-with-saved so the user can actually see that the substitution has happened.
  • Substitution target is visually highlighted in the prompt preview — yellow <mark> on the substituted token, muted grey on the empty placeholder. Atum dark-mode styles included.

📦New Features

  • update_menu_item now writes to the menu item's params blob. Named SEO args (browser_page_title, meta_description, meta_keywords, robots, page_heading, show_page_heading, page_class_sfx, menu_anchor_title, secure) plus generic params_set / params_unset escape hatches. Merges into existing params — keys you don't touch are preserved byte-for-byte. robots enum-validated. Closes the home-page-meta-description gap.
  • get_schemaorg_site_profile and set_schemaorg_site_profile — typed wrappers for plg_system_schemaorg's site-wide Organization/Person profile. Knows the actual four-key shape the plugin reads (baseType, name, image, socialmedia) — not the wrong Organization_* flat keys. Hard-enforces base_type lowercase — capital-O Organization silently kills the plugin's entire @graph including per-article schemas. Tool descriptions point at 4SEO's Business Profile for full LocalBusiness coverage since this plugin doesn't support telephone/address/geo.

v1.7.3 2026-05-11

🐛Bug Fixes

  • CRITICAL: Postflight failed on Joomla 6 with Class "Joomla\CMS\Filesystem\File" not found. That class was removed in J6 (deprecated in J4). On J6 sites, the autoload-cache clear AND the plugin-enabling step were both silently skipped, leaving the Web Services plugin disabled and the MCP route 404'ing. Fixed by replacing the one File::delete() call with plain @unlink() — no Joomla classes involved. If you installed v1.6.x–v1.7.2 on a Joomla 6 site, install v1.7.3 over top and the plugins will get re-enabled automatically.

v1.7.0 2026-05-11

Refactor of the 4SEO add-on now that we have 4SEO v6.12.0's full source available locally for design reference. Adds 5 typed tools so an agent can manage per-page meta and 4SEO config without knowing about the three-layer envelope or size-based column routing. The generic CRUD escape hatches stay. Tool count 73 → 78.

📦New Features

  • list_4seo_meta_overrides — typed audit query over #__forseo_custom_meta. Decodes the data JSON for you and surfaces custom_title / custom_description / custom_robots / custom_canonical alongside the row.
  • get_4seo_meta_override — read one row by content_id, joomla_params, or article_id. Returns all three layers (platform / auto / custom) separately.
  • set_4seo_meta_override — three-layer-aware upsert. Just supply title / description / robots / canonical; the tool sets them in the custom layer and bumps status flags to 2 (custom). Preserves platform & auto layers untouched. Creates the row if missing.
  • clear_4seo_meta_override — reset_to_auto (default; wipes custom layer, status flags back to 0) or delete_row (hard remove).
  • set_4seo_config — typed counterpart to get_4seo_config. Auto-encodes value_object to JSON; auto-routes >16000-byte payloads to the large_value column. Upserts by (scope, key).

🛡️Design notes

  • 4SEO has no standard Joomla Model classes — the "Option B" path (call into Weeblr's PHP models) is unavailable. We're on Option A (DB-direct) but with typed wrappers built from the canonical install.sql schema, giving the agent the ergonomics of B without the models.
  • Generic query_4seo_table / insert_4seo_row / update_4seo_row / delete_4seo_row escape hatches retained for the 20+ tables that still need typed wrappers (rules, sitemaps, perf data, GSC, referrers, errors, links, images). They're now last in the registration so the agent reaches for typed tools first.

v1.5.1 2026-05-09

Patch release responding to v1.5.0 test feedback. Fixes the JSON-RPC corruption class of bug structurally and unblocks editing legitimately user-editable locked plugins via set_plugin_params.

🐛Bug Fixes

  • Critical: stray PHP output corrupted JSON-RPC envelopes. v1.5.0's fetch_rendered_url cast an array-valued Content-Type header to string, emitting a PHP warning that prefixed the JSON response and broke parsing in any MCP client. Wrapped McpController::handle() in ob_start() and discard the buffer before emitting JSON — defends every tool, present and future, from this whole class of bug. Local cause in FetchRenderedUrlTool also fixed (multi-value headers joined with , ).
  • set_plugin_params couldn't modify legitimately user-editable locked plugins like plg_system_schemaorg. Added allow_locked: true flag to override the lock guard explicitly; protected: true plugins still get a hard refusal.

🔧Improvements

  • fetch_rendered_url returns jsonld_types — flat, dedup'd, sorted array of every @type value across all JSON-LD blocks (recursively walking @graph). "Did my X type land?" becomes a single inclusion check.
  • set_article_custom_jsonld and set_article_custom_jsonld_bulk descriptions now warn against wrapping the supplied JSON-LD in a top-level @graph — Joomla 5.1+ merges each block into the page's existing @graph automatically.

v1.5.0 2026-05-09

Feature release responding to the v1.4.x test feedback. Adds five new tools, total counts on paginated responses, and a token-substitute UI on the dashboard so non-technical users can copy a fully-ready prompt without manual editing. Tool count: 67 → 72.

📦New Features

  • get_plugin_params / set_plugin_params — generic plugin params read/write via direct DB. Unlocks site-wide schemaorg config (folder=system, element=schemaorg), router options, third-party plugin config, anything on any plugin's Options screen. Refuses protected/locked core plugins.
  • fetch_rendered_url(path, extract_jsonld?) — fetches a rendered page from the SAME Joomla site (same-origin only — no SSRF). Optional JSON-LD extractor pulls every <script type="application/ld+json"> block out of <head> and returns them parsed — closes the verification loop for Schema.org workflows.
  • set_article_custom_jsonld_bulk — bulk variant for sites with hundreds of articles. Per-item independent, capped at 500 per call.
  • get_4seo_config (4SEO add-on) — reads #__forseo_config where the real settings live (the Joomla extensions row's params field is near-empty for 4SEO).
  • total field on paginated responses — list_articles, list_articles_with_schema, list_users, and query_4seo_table now return the count across the whole filtered set so the agent knows when pagination is done without poll-till-empty.

🔧Improvements

  • Token-substitute UI on the dashboard's prompt tab. Optional input field for the user's token; when set, the Copy button substitutes the <PASTE YOUR JOOMLA API TOKEN HERE> placeholder before copying. Token never leaves the browser. Button label flips to "Copied with token included!" so the user knows substitution happened.

v1.0.0 2026-04-25

Initial release with 51 built-in tools across 11 Joomla admin domains.

📦New Features

  • MCP endpoint: Streamable-HTTP MCP server at /api/index.php/v1/mcp, authenticated by Joomla API token (X-Joomla-Token or Authorization: Bearer).
  • JSON-RPC 2.0 server: initialize, notifications/initialized, ping, tools/list, tools/call. Single messages and batches supported.
  • ACL gating: csmcpforj.use (read) and csmcpforj.write (write) actions. Super Users / Administrator / Manager pass automatically.
  • Tool registry: Plugins extend the surface by subscribing to the onCsMcpRegisterTools event.
  • Admin dashboard: Endpoint URL, ready-to-paste client config, permissions table, and grouped tool list.
  • Built-in tool set (51 tools): Articles, Categories (any extension), Tags, Menus, Users & Access, Modules, Extensions/Plugins, Templates, Languages, Custom Fields, System.

🔍Security

  • Per-tool permission gate runs before every tools/call execution.
  • access.xml ships with the component so per-action permissions appear in System → Permissions immediately after install.
  • All writes route through the corresponding com_* Administrator model so workflow, asset, and ACL side effects stay consistent.
  • Database access uses quoteName() and parameterised values throughout — no string concatenation into SQL.
  • delete_user refuses to delete the calling user.
  • set_extension_enabled refuses to touch protected or locked core extensions.
  • get_site_info deliberately omits secrets (DB password, mailer credentials, captcha keys).
